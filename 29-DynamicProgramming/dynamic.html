Dynamic Programming

Dynamic Programming is a method for solving a complex problem by breaking it down into a collection of simpler subproblems, solving each of those subproblems just once, and storing their solutions.

Dynamic Programming isn't for every problem, but when it works, it can really speed things up.

The name itself is a little misleading and infact wasn't defined with computers in mind. It isn't trying to work on a million inputs - it just sounds that way.

Dynamic Programming really only works on problems with optimal substructure and overlapping subproblems.

Overlapping Subproblems

A problem is said to have overlapping subproblems if it can be broken down into subproblems which are reused several times.

So there needs to be a repetition and they need to overlap. Something like mergesort, which breaks down problems and then rebuilds them would NOT be overlapping enough to work with dynamic programming.

Optimal substructure

A problem is said to have optimal substructure if an optimal solution can be constructed from optimal solutions of its subproblems.

Things like shortest path are NOT optimal substruture because the cheapest flight to SEA might not be included in the cheapest flight to Fairbanks.

Memoization

A key part of Dynamic programming is using past knowledge to make solving a future problem easier.

Memoization is storing the results of expensive function calls and returning the cached result when the same inputs occur again.

Check out the demo. Makes a huge difference. Gets down to O(n) time.

Tabulation

Tabulation is storing the result of a previous result in a 'table', usually an array. Usually done with iteration and has better space complexity than memoization. This is the 'bottom up' type approach.

Because Tabulation is iterative, it won't run into a stack overflow error like a recursive, memoized function might.
